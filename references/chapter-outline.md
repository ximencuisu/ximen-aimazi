# 章节细纲的分批生成、迭代返修与逻辑审核

大纲确认后，不默认一次性生成全书细纲。更适合长篇网文的做法，是按批次滚动生成、反复返修、冻结后再写正文。

---

## 总流程

```text
大纲确认
  ↓
卷级规划
  ↓
批次细纲生成（当前批次）
  ↓
逻辑审核
  ↓
人工介入决策（方案选项 / 自定义）
  ↓
返修 / 回退上游
  ↓
冻结当前批次
  ↓
【门控检查：大纲存在 + 细纲已冻结】
  ↓ (不通过 → 拒绝进入正文)
进入正文
  ↓
【记忆写入：CHARACTERS + PLOT_POINTS + SUSPENSE + CHAPTERS + SESSION】
  ↓
一致性审核（可跳过）
  ↓
润色与评分
  ↓
【记忆写入：SESSION + ERRORS（如质量不达标）】
  ↓
开启下一批次细纲
```

> 默认只处理"当前批次 + 下一批缓冲"，而不是一口气写满全书。
> 
> **大纲/细纲修改**：修改后必须重新走审核冻结流程，并同步写入记忆。详见 SKILL.md"修改流程"章节。

---

## 第一部分：为什么要按批次做细纲

长篇连载里，细纲往往会被以下情况反复冲击：

- 用户临时换卖点或换卷方向
- 角色表现和最初设想不同
- 爽点兑现位置需要前移或后移
- 中途发现伏笔太重、节奏太拖、钩子不够
- 正文写出来后，某些桥段需要回调

所以细纲不该被当成"一次性总图"，而应该被当成"可滚动维护的中控层"。

## 第二部分：批次规划规则

### 批次大小建议

| 场景 | 建议每批章数 |
|------|-------------|
| 开篇前 20 章 | 3-5 章 |
| 稳定推进期 | 5-8 章 |
| 大高潮 / 换地图 / 换卷 | 2-4 章 |
| 风险提示 | 单批不建议超过 10 章 |

### 规划原则

1. **先卷级，后批次，再到单章**：先锁卷目标，再写批次。
2. **硬锚点先锁死**：关键机缘、卷末状态、重大反转、主线伏笔先固定。
3. **软空间留给返修**：桥段顺序、配角出场、单章高潮位置可调整。
4. **一批只承诺一批的事**：当前批次只对当前批次的兑现负责。
5. **没冻结，不进正文**：未冻结细纲只算草稿，不算正式施工图。

### 卷级规划模板

```markdown
## 卷级规划

| 卷 | 章节范围 | 卷目标 | 卷末状态 | 关键伏笔 |
|----|---------|--------|---------|---------|
| 第一卷 | 1-30 | 立住主角与修炼路线 | 主角拿到第一次真正的立身资本 | 主线伏笔A |
| 第二卷 | 31-60 | 扩大地图与势力冲突 | 主角被迫站队 | 主线伏笔B |
```

### 批次规划模板

```markdown
## 批次规划

| 批次 | 章节范围 | 本批任务 | 必须兑现 | 章末钩子 | 风险点 | 状态 |
|------|---------|---------|---------|---------|-------|------|
| 批次1 | 1-5 | 开书钩子+第一次机缘 | 金手指亮相、主角第一次反压 | 引出更大威胁 | 节奏过慢 | 草拟 |
| 批次2 | 6-10 | 扩大战果+引入对手 | 第一次公开打脸 | 引出阶段目标 | 角色动机薄弱 | 待开始 |
```

### 冻结规则

当前批次只有满足以下条件，才允许冻结：

- 无硬逻辑矛盾
- 本批次承诺的爽点 / 信息已经兑现
- 本批次结尾能自然牵引下一批
- 人工介入节点已经完成，用户已做出选择
- 如返修会影响上一批，必须先回改上一批再冻结当前批

## 第三部分：批次细纲模板

```markdown
# 批次细纲：第001-005章（V1）

- **状态**：草稿 / 待审核 / 已冻结
- **本批任务**：开书钩子 + 第一次机缘兑现
- **承前**：接大纲第一卷开场目标
- **启后**：为第006-010章的外部冲突升温做准备
- **硬锚点**：金手指亮相、第一次反压、结尾引出宗门试炼
- **可调整空间**：桥段顺序、配角出场先后、单章高潮位置

## 第001章 [章节名]

### 本章定位
- **章节类型**：剧情推进 / 战斗 / 转折 / 过渡
- **本章任务**：本章必须完成什么
- **与上章衔接**：如何接续上一章
- **本章兑现**：本章交付给读者的爽点或信息
- **章末钩子**：把读者带去下一章的悬念

### 剧情推进
- **开场**：
- **发展**：
  1. ...
  2. ...
  3. ...
- **高潮 / 转折**：
- **收束**：

### 人物 / 关系变化
- **出场人物**：
- **状态变化**：
- **关系变化**：

### 世界 / 资源 / 伏笔
- **地点**：
- **获得 / 消耗**：
- **埋设**：
- **回收**：

### 返修接口
- **可微调**：
- **不可轻改**：
- **风险提示**：
```

### 生成要求

1. 每章都要说明"本章任务"和"本章兑现"。
2. 每章都要有章末钩子，哪怕是过渡章也要给出牵引。
3. 批次顶部必须写清本批任务、硬锚点、可调整空间。
4. 默认不跳过返修接口。细纲多轮迭代时，这一栏非常有用。

## 第四部分：逻辑审核、人工介入与冻结

### 审核维度

| 维度 | 检查项 |
|------|--------|
| 时间线 | 章节衔接、时间跳跃、成长速度 |
| 空间线 | 地点转移、行动距离、地图切换成本 |
| 人物线 | 存活状态、等级匹配、关系变化、动机连贯 |
| 设定线 | 力量体系、物品来源、社会规则、系统边界 |
| 伏笔线 | 埋设、回收时机、后续可承接性 |
| 读感线 | 章节功能是否清楚、爽点是否兑现、章末钩子是否有效 |

### 审核流程

```text
1. 读取：上一批已冻结细纲 + 当前批次细纲 + 当前卷目标
2. 逐章检查当前批次
3. 跨章节检查：至少回看前 3-5 章与下一批目标
4. 判断问题级别：
   - 微调
   - 当前批重做
   - 回改上一批
   - 回改大纲
5. 生成 2-4 个相应方案 + 1 个自定义选项
6. 用户选择：直接冻结 / 某个方案 / 自定义要求
7. 按选择修改后重新审核
8. 满足冻结条件后，更新冻结清单
9. 只有已冻结章节才允许进入正文
```

### 审核结论

- `通过并冻结`：可直接进入正文
- `小修后冻结`：修正局部问题后即可冻结
- `退回当前批次`：当前批次结构不稳，需要重做
- `上游返工`：问题来自前批次或大纲，必须先回改上游

### 人工介入规则

1. 每次批次审核后，默认进入人工介入节点。
2. AI 必须给出 2-4 个相应方案，不超过 4 个，避免选择过载。
3. 推荐方案必须放第一位，并说明为什么推荐。
4. 每个方案都要写清：
   - 改动
   - 好处
   - 代价
   - 影响范围
5. 最后固定保留 `自定义` 选项。
6. 如果用户说"你来定"，才允许按推荐方案直接执行。

### 人工介入模板

```markdown
## 当前批次：第001-005章（V2）
## 当前判定：小修后可冻结
## 推荐方案：A

### 方案 A：稳妥小修版（推荐）
- **改动**：只修正第002章钩子偏弱、第004章动机不清
- **好处**：最稳，不伤主线结构
- **代价**：爽点提升有限
- **影响范围**：当前批次

### 方案 B：爽点前移版
- **改动**：把第一次反压从第005章前移到第003章
- **好处**：开篇更抓人
- **代价**：后两章需要重排承接
- **影响范围**：当前批次 + 下一批开场

### 方案 C：人物关系加压版
- **改动**：增强对手压迫和师门关系冲突
- **好处**：人物张力更强
- **代价**：前期节奏会更沉一点
- **影响范围**：当前批次

### 自定义
- 你可以直接指定：
  - 哪一章保留不动
  - 哪个爽点前移或后移
  - 哪条人物线要加强或删掉
  - 是否需要回改上一批或大纲
```

### 审核检查表

```markdown
### 必检项
- [ ] 与上一批已冻结章节衔接自然
- [ ] 时间线与空间线无硬伤
- [ ] 人物动机、关系、战力不崩
- [ ] 本批次承诺的爽点 / 信息已兑现
- [ ] 章末钩子能牵引下一章
- [ ] 伏笔埋设与回收有后续承接
- [ ] 如需返修，已标清影响范围

### 跨批次检查
- [ ] 与前 3-5 章无冲突
- [ ] 与当前卷目标无冲突
- [ ] 不会迫使下一批次进入死路
```

## 第五部分：输出文件

### 推荐文件结构

```text
output/
├── 大纲.md
├── 细纲.md
├── 细纲迭代记录.md
├── 细纲干预决策.md
├── 细纲冻结清单.md
├── 批次细纲/
│   ├── 第001-005章_v1.md
│   └── 第001-005章_v2.md
└── 逻辑审核报告.md
```

### 文件职责

- `output/细纲.md`
  - 细纲主控文件
  - 记录卷结构和已冻结批次摘要
- `output/批次细纲/第001-005章_vN.md`
  - 当前批次或历史版本的细纲稿
- `output/细纲迭代记录.md`
  - 记录每次返修原因、结论和影响范围
- `output/细纲干预决策.md`
  - 记录每轮审核后提供的方案、推荐项和用户选择
- `output/细纲冻结清单.md`
  - 记录哪些章节已冻结、哪些还不能写正文
- `output/逻辑审核报告.md`
  - 当前批次审核结果

```markdown
# 细纲迭代记录

|------|------|------|------|----------|----------|----------|----------|
| 初始化 | 第001-005章 | V0 | 待生成 | — | — | 建立跟踪文件 | output/细纲.md |
```

## 第六部分：示例

### 场景：玄幻 / 修仙长篇

```text
用户：大纲已确认，开始做细纲。

AI：
1. 先规划第一卷，把第1-30章拆成 6 个批次
2. 先只生成第001-005章细纲 V1
3. 审核发现问题：
   - 第002章信息铺垫过多，钩子太弱
   - 第005章才第一次反压，兑现偏晚
4. 返修为 V2：
   - 把关键机缘提前到第001章结尾
   - 把第一次反压提前到第003章结尾
5. 进入人工介入节点，给出 3 个方案 + 自定义
6. 用户选择"方案 B：爽点前移版"
7. 按方案 B 改成 V2，再次审核通过，冻结第001-005章
8. 正文只写第001-005章
9. 再开启第006-010章细纲
```

## 第七部分：常见问题

### Q1：什么时候需要解冻已经冻结的批次？

以下情况可以解冻：

- 新增设定会直接冲掉旧批次
- 正文实写后发现旧批次承接不了
- 用户明确要求改主线方向

一旦解冻，必须在 `output/细纲迭代记录.md` 里记一条"解冻返修"，并暂停受影响章节的正文。

### Q2：要不要先写完整本书的细纲再开正文？

默认不要。对长篇连载来说，先锁卷目标，再滚动做当前批次，通常更稳。

### Q3：批次之间怎么保持连续？

每次生成新批次前，至少读取：

- 前一个已冻结批次
- 当前卷目标
- `output/细纲冻结清单.md`
- `.learnings/` 活跃部分

### Q4：什么叫"硬锚点"？

硬锚点是不能随便改的位置，比如：

- 主角第一次真正反压
- 关键外挂亮相
- 卷末状态
- 主线大伏笔首次落地

这些点一旦改动，通常要连带回看前后批次。

### Q5：如果我不想选现成方案怎么办？

直接用 `自定义`。你可以一句话指定：

- "保留第1、2章不动，把第一次反压提前到第3章结尾"
- "删掉师姐线，改成师门任务线"
- "不要回改大纲，只允许在当前批次内微调"
